# behavior.md — 기본 행동 규칙 (항상 적용)

> Karpathy 4원칙(LLM 코딩 실수 줄이기) + 본 프로젝트의 작업 위생.
> 출처: https://github.com/multica-ai/andrej-karpathy-skills (MIT)

## 0. 기획 방법론 5대 원리 (모든 문서·기능에 항상 적용)
- 단일 진실 / ID로 추적 / 모든 상태 / Out of scope / Given–When–Then.
- 상세·셀프 검토 프롬프트는 `rules/methodology/README.md`.

## 1. 코드 전에 생각한다 — 가정·트레이드오프를 드러낸다
- 추측해서 채우지 말 것. 적혀 있지 않으면 "적혀 있지 않음"이라 표시하고 사람에게 묻는다.
- 해석이 둘 이상이면 조용히 고르지 말고 모두 제시하고 사람에게 고르게 한다.
- 더 단순한 길이 보이면 그렇게 말한다. 필요하면 요청을 되돌려 묻는다.
- 작업 전 관련 docs/ 를 먼저 읽는다 (REQ → SCR → POL → TC).

## 2. 단순함을 먼저 — 요청한 만큼만
- 요청 이상 만들지 않는다. 투기성 추상화·옵션·"유연성"·미래 대비 설정은 금지.
- 일어날 수 없는 시나리오에 대한 방어 코드를 만들지 않는다.
- 200줄로 쓴 것이 50줄로 끝난다면 다시 쓴다.
- 자문: "시니어가 보면 과하다고 할까?" 그렇다면 줄인다.

## 3. 외과적으로 바꾼다 — 범위 밖은 건드리지 않는다
- 작업 범위 밖의 코드·주석·포맷·네이밍은 손대지 않는다.
- 깨지지 않은 것은 리팩터링하지 않는다. 기존 스타일과 glossary.md 표기를 그대로 따른다.
- 내 변경이 만든 잔재(미사용 import/변수/함수)만 제거한다. 기존 dead code 는 보고만 하고 두지 않는다.
- 모든 변경 라인은 사용자의 요청까지 직선으로 추적되어야 한다.

## 4. 검증으로 끝낸다 — 성공 기준을 먼저 못 박는다
- 작업 시작 전 "성공이란 무엇인가"를 한 줄로 정의한다. "유효성 추가해" → "잘못된 입력 테스트가 통과한다".
- 코드를 바꾸면 관련 테스트를 함께 추가/갱신한다. "되는 것 같다"가 아니라 테스트로 증명한다.
- 멀티스텝 작업은 단계마다 검증 체크를 단다: `1. [단계] → 검증: [확인]`.
- 작업 후 DEVLOG.md 에 무엇을·왜를 한 줄 남긴다.
