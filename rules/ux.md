# UX 원칙 (모바일)

UI를 다듬거나 디자인 결정을 검증할 때 읽는다. 시각 토큰은 [styling.md](styling.md), View 구조는 [ui.md](ui.md).

모바일 UX 지침입니다. 원칙은 보편적이며, 구현은 [네이티브](iOS SwiftUI / Android)와 [웹·하이브리드](PWA·WebView)로 구분해 표기합니다. 수치 근거: Apple HIG · Material Design 3 · WCAG 2.2.

## 1. 터치는 손가락 크기로 설계한다
탭 가능한 요소는 손가락으로 오동작 없이 누를 수 있어야 한다.

- 최소 타깃: 44pt(iOS) / 48dp(Android). WCAG 최소는 24px(권장 44px). 크기뿐 아니라 요소 간 **간격(8pt 이상)**도 확보한다.
- 디자인상 작으면 보이지 않는 패딩으로 터치 영역을 확장한다 (아이콘 버튼 특히 주의).
- [네이티브] SwiftUI `frame(minWidth:minHeight:)`·`contentShape`로 히트 영역 확보. [웹·하이브리드] `min-width/height` + padding으로 확장.

**점검**: 가장 작은 버튼이 44pt/48dp 이상이고, 인접 버튼과 충분히 떨어져 있는가?

## 2. 모든 동작에 즉각 피드백을 준다
탭에 아무 반응이 없으면 사용자는 고장났다고 느낀다.

- 누르는 즉시 눌림 상태(밝기·스케일 변화)와, 필요 시 햅틱으로 반응한다.
- [네이티브] 버튼 pressed 상태 + 햅틱(iOS UIImpactFeedbackGenerator / SwiftUI `.sensoryFeedback`). [웹·하이브리드] 모바일엔 호버가 없으므로 hover 의존 금지 → `:active` 스타일로 즉시 피드백.

**점검**: 모든 탭에 0.1초 안에 시각·촉각 반응이 있는가?

## 3. 체감 속도를 설계한다
실제 속도보다 '느껴지는 속도'가 경험을 좌우한다.

- 로딩은 빈 화면·스피너 대신 콘텐츠 형태의 스켈레톤으로 채운다.
- 좋아요·토글처럼 가벼운 동작은 낙관적 UI로 — 즉시 성공 상태로 바꾸고, 백엔드 실패 시에만 원래 상태로 롤백한다.
- 응답 지연 임계를 의식한다: 즉각 <0.1s · 끊김 없는 흐름 <1s · 주의 유지 <10s.
- [네이티브] redacted/placeholder 뷰. [웹·하이브리드] 스켈레톤 컴포넌트.

**점검**: 데이터 로딩 중 화면이 비거나 멈춰 보이지 않는가?

## 4. 로딩만이 아니라 '모든 상태'를 디자인한다
한 화면에는 정상 외에도 로딩·빈 상태·에러·성공이 있다.

- 빈 상태에는 다음 행동을 제시한다 ("아직 없습니다"로 끝내지 않는다).
- 에러에는 무엇을 하면 되는지를 알려준다 ("오류가 발생했습니다"·에러 코드만 띄우지 않는다).

**점검**: 데이터 0건 · 네트워크 실패 · 권한 거부 상태를 각각 그렸는가?

## 5. 기기의 물리 구조를 존중한다
노치·펀치홀·홈 인디케이터에 콘텐츠가 가리지 않게 한다.

- 헤더와 하단 바는 safe area를 침범하지 않는다. 화면 높이는 고정하고 콘텐츠 영역만 내부 스크롤한다.
- [네이티브] SwiftUI가 safe area를 대부분 자동 처리(`.safeAreaInset`, `ignoresSafeArea`는 신중히 사용). [웹·하이브리드] `env(safe-area-inset-*)`를 패딩으로 직접 계산하고, 바운스/오버스크롤은 고정 레이아웃으로 차단한다.

**점검**: 노치 기기와 홈 인디케이터 기기에서 잘리는 요소가 없는가?

## 6. 접근성을 기본값으로 둔다
처음부터 모두가 쓸 수 있게 만든다 — 나중에 붙이지 않는다.

- 글자는 고정 크기가 아니라 Dynamic Type로 스케일되게 한다 (최소 11pt, 본문 17pt 권장).
- 화면 리더 라벨을 제공하고, 색만으로 정보를 전달하지 않으며, 대비는 4.5:1(큰 텍스트·UI 요소는 3:1)을 지킨다.
- **동작 줄이기(reduce motion)**와 다크 모드를 지원한다.
- [네이티브] Dynamic Type(`.font(.body)`) · VoiceOver(`accessibilityLabel`) · `@Environment(\.accessibilityReduceMotion)`. [웹·하이브리드] 상대 단위(rem) · `prefers-reduced-motion` · `prefers-color-scheme` · ARIA 라벨.

**점검**: 시스템 글자 크기를 최대로 키워도 레이아웃이 깨지지 않는가?

## 7. 일관성과 플랫폼 관습을 따른다
OS의 관습을 거스르지 않는다. 같은 것은 어디서나 같게.

- 성공·경고·오류 등 의미론적 컬러를 시스템 전체에서 일관되게 쓴다.
- 내비게이션은 예측 가능하게 한다 (되돌아가기 명확, iPhone 탭 바는 5개 이하).
- [네이티브] 시스템 컴포넌트·SF Symbols·시맨틱 컬러를 우선한다. [웹·하이브리드] 디자인 토큰으로 색·간격을 통일하고 플랫폼별 관습을 존중한다.

**점검**: 새 사용자가 학습 없이 다음 행동을 예측할 수 있는가?

## 8. 콘텐츠 우선, 군더더기 최소
사용자의 콘텐츠가 먼저 보이고, UI는 필요할 때 드러난다.

- 화면을 열면 '내 것(콘텐츠)'이 먼저 보여야 한다 — 앱의 인터페이스가 아니라.
- 애니메이션은 목적이 있고 빠르게. 입력은 최소화한다 (적절한 키보드 타입·인라인 검증).

**점검**: 첫 화면에서 사용자가 곧장 핵심 콘텐츠·행동에 닿는가?
