## ① PRD — 서비스 기획서
- **기본 정의**: 한 줄 정의 / 배경·문제 / 타겟 / 핵심 가치·차별점 / 마일스톤 / 제약·전제
- **범위 (In / Out / Backlog)** — ★ **Out of scope가 진짜 핵심** (AI가 범위 넘지 않게)
- **KPI**: 명 · 측정 방법 · 목표 수치 · 기간 (4칸 다 있어야 검증 가능. '많이 쓰이게'는 KPI 아님)
- **비기능**: 성능 · 접근성 · 보안 · 다국어 · 오프라인 · 데이터 보관/삭제 · 로깅 · 개인정보 (가장 자주 빠짐)

## ② POL — 정책 정의서
| 정책 ID | 분류 | 정책 내용 | 적용 조건 | 예외 처리 | 사유 | 적용 시점 |

'사유'와 '적용 시점'을 함께 — 나중에 안전하게 풀거나 바꿀 수 있게. 이유 없는 금지는 족쇄가 된다.
분류 예: 입력 검증 / 권한·접근 / 노출·정렬 / 상태 전이 / 예외·오류.

## ② glossary — 용어·데이터 정의서 ★ 단일 진실
| 용어(한글) | 정의 | 코드/영문명 | 타입·범위 | 사용 화면 | 금지 표현 |

★ **다른 모든 문서는 이 표를 '가리킬' 뿐. 값을 다시 쓰지 않는다.** '타입·범위'가 단일 진실의 핵심.

## ③ REQ — 요구사항 정의서
| 요건 ID | 구분 | 요구사항명 | 상세 | 우선순위 | 관련 화면·정책 | 검증(TC) |

- 우선순위 = MoSCoW: Must / Should / Could / Won't
- 비기능(성능·접근성·보안)도 같은 표에
- '관련 화면·정책' 비어 있으면 미정 — 마감 전 채움

## ③ IA — 정보구조도
| Depth 1 | Depth 2 | Depth 3 | 화면 ID | 설명 |

화면 ID는 SCR과 동일. 트리/사이트맵 다이어그램을 함께 두면 이해가 빠름.

## ③ FLOW — 사용자 플로우
| Step | 구분 | 사용자 행동 | 시스템 반응 | 다음 화면 | 관련 정책 |

'구분'을 **정상 / 분기 / 예외** 3분류로 — 빠진 예외가 한눈에 보임. 예외는 POL-###과 연결.

## ③ SCR — 화면 정의서 ★ 상태 매트릭스
- **기본 정보**: ID/명 · 목적 · 진입·다음 · 관련 REQ·POL · 레이아웃
- **구성요소**: No · 영역 · 설명 · 동작·인터랙션 · 데이터·표시 규칙
- **★ 상태 매트릭스 (6개)** — 적용 불가는 `N/A` 명시

| 상태 | 표시 내용 | 사용자 행동 유도 | 관련 정책·요건 |
| --- | --- | --- | --- |
| 정상 |  |  |  |
| 로딩 |  |  |  |
| 빈(empty) |  |  |  |
| 에러 |  |  |  |
| 권한 없음 |  |  |  |
| 오프라인 |  |  |  |

> '오류가 발생했습니다'로 끝내지 말 것 — AI는 그 문구를 그대로 가져다 쓴다. 각 상태에 '사용자가 무엇을 하면 되는지' 함께.

## ④ TC — 테스트 시나리오
| TC ID | 대상 REQ | Given (전제) | When (행동) | Then (결과) | 관련 정책 |

★ Given-When-Then은 자동 테스트로 그대로 변환 가능해야 함. 비기능(성능·접근성)도 같은 형식.

```
TC-099 (성능)
  Given  보통 4G 환경
  When   '검색' 버튼을 누른다
  Then   첫 결과가 0.3초 안에 노출된다

TC-100 (접근성)
  Given  VoiceOver ON
  When   결과 카드에 포커스가 간다
  Then   상품명·가격·평점이 순서대로 읽힌다
```

## ④ 릴리스 / 버전 노트
| 버전 | 일자 | 유형 | 변경 내용 | 이유·배경 | 관련 문서 |

'무엇을'만이 아니라 **'왜 그렇게 결정했나' + '다음에 할 것'** 을 남기면, 다음 세션 AI와 새 멤버에게 최고의 인수인계가 됨.
