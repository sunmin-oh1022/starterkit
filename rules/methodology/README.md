# rules/methodology/ — 기획자 × AI 페어 작업 방법론

> '어떻게 일하는가'의 단일 진실. 매 세션 진입 시 AGENTS.md → rules/behavior.md 를 통해 5대 원리가 자동 적용된다.
> 9개 기획 문서별 작성 칸·예시는 → [templates.md](./templates.md)

## 한 문장 요약
> **"AI에게 일을 시킨다"가 아니라 "AI가 잘 일할 수 있는 환경(규칙·맥락·검증·기억)을 미리 설계한다."**

사람 동료는 막히면 묻지만 AI는 묻지 않고 빈 곳을 그럴듯하게 메운다(환각). 그래서 기획자의 새 역할 = **'읽힐 문서'가 아니라 '실행될 문서'** 를 쓴다.

## 5대 원리 (모든 문서·기능에 항상 적용)
1. **단일 진실** — 한 사실은 한 곳에만. 같은 값이 두 문서에 보이면 한쪽은 ID 참조로.
2. **ID로 추적** — `REQ-### → SCR-### → TC-###`, 정책 `POL-###`, 플로우 `FLOW-###`.
3. **모든 상태** — 정상 외에 로딩·빈·에러·권한·오프라인. 적용 불가는 `N/A` 명시.
4. **Out of scope** — '이번엔 안 함' 명시. 좁힐수록 빠름.
5. **Given–When–Then** — '빠르게·자연스럽게·직관적으로' 금지. 숫자·동작으로.

> 셀프 점검 신호: 막연한 표현이 보이면 멈추고 숫자/동작으로 / 같은 값이 두 곳에 보이면 한쪽을 ID 참조로.
>
> 위 5대는 **기획** 원리. **코드 작성** 시 LLM 룰은 별도로 둔다 — 예: `rules/behavior.md` 에 Karpathy 4원칙(생각→단순함→외과적 변경→검증)을 흡수해 상보적으로 운용.

## 6 Phase (기능·버전마다 1→5 반복, Phase 0은 프로젝트당 1회)
| Phase | 이름 | 핵심 질문 | 산출물 |
| --- | --- | --- | --- |
| 0 | 환경 셋업 (Day 0) | 어떤 규칙 위에서 일하나 | 저장소·진입점·rules/·1초 검증 |
| 1 | 기획 문서 작성 | 무엇을·왜 만드나 | PRD · POL · glossary · REQ · SCR · TC |
| 2 | 셀프 검토 | 구멍·모순 없나 | AI 검토 결과 + 마감 체크 통과 |
| 3 | 핸드오프 | AI에 어떻게 넘기나 | 한 줄 작업 지시 |
| 4 | 구현·검증 루프 | 맞게 됐나 | 통과한 테스트 + 정합성 검증(`scripts/check-docs.sh`) + 실기기 확인 |
| 5 | 기록·반복 | 어떻게 기억하나 | 커밋 · DEVLOG · HISTORY |

## 문서 4단 체계 (위→아래 = 넓은 맥락 → 구체 검증)
| 단계 | 문서 | 한 줄 |
| --- | --- | --- |
| ① 상시 브리프 | PRD | 무엇을·왜·누구를 위해 |
| ② 가드레일 | POL · glossary | 지킬 규칙·예외 / 표기 통일 (단일 진실) |
| ③ 가이드 | REQ · IA · FLOW · SCR | 실행 지침 + 상태 매트릭스 |
| ④ 검증·기록 | TC · DEVLOG · HISTORY | GWT로 증명 + 결정 보존 |

규모가 작으면 ①③④만으로도 충분. 정책·예외가 많을수록 ②의 비중↑.

## 폴더 구조 (rules / docs 분리 — 메타 규칙 vs 산출물)
```
project/
├── AGENTS.md (또는 CLAUDE.md)         ← 얇은 진입점 (200줄 이하)
├── rules/                             ← "어떻게 일하나" — 메타 규칙
│   ├── behavior.md · boundaries.md    (항상 @import)
│   ├── architecture.md · commands.md · domain.md · ui.md · ux.md  (작업별)
│   └── methodology/                   ← 이 폴더 — 기획·작업 방법론
├── scripts/                           ← 1초 검증·정합성 자동화
│   └── check-docs.sh                  (참조 경로·자동 로드 체인·테스트 한 번에)
└── docs/                              ← "무엇을 만드나" — 산출물
    ├── product/        PRD · glossary · personas
    ├── requirements/   REQ · flows/FLOW-###
    ├── policies/       POL
    ├── screens/        SCR-### (★ 상태 매트릭스)
    └── acceptance/     TC
```

도구별 진입점: Claude Code → `CLAUDE.md`, Codex → `AGENTS.md`. 한 벌만 관리하려면 `ln -s AGENTS.md CLAUDE.md`.

## Phase 0 — Day 0 체크리스트 (프로젝트당 1회)
- [ ] AI 코딩 에이전트 설치·로그인
- [ ] Git 비공개 저장소 + 첫 커밋
- [ ] 프로젝트를 'Core(순수 로직) / App(UI)' 두 레이어로
- [ ] 진입점(AGENTS.md / CLAUDE.md) — 항상 읽을 규칙만 import + Rule Index
- [ ] rules/ 모듈 — 항상(behavior·boundaries) / 작업별(나머지)
- [ ] 가드레일: 금지 + 대안 + 이유
- [ ] 번호 가이드 #1, #2 … ("#3 위반"처럼 짧게 지적용)
- [ ] 1초 검증 명령 확정 (예: `swift test`) + 정합성 검증 스크립트(`scripts/check-docs.sh`)
- [ ] 사람/AI 검증 경계 명문화
- [ ] DEVLOG.md / HISTORY.md / SHARE.md 생성

> 위는 일반 템플릿. 각 프로젝트는 자기 진행 상태를 별도로 추적 (예: 저장소 루트의 `CHECKLIST.md` — 항목 옆 `[done]` 로 완료 표시).

## Phase 2 — 셀프 검토 프롬프트 (마감 전, 그대로 복사)
```
다음 기획 문서들을 읽고 점검해줘. 추측해서 채우지 말고,
실제 적혀 있지 않은 것은 '적혀 있지 않음'으로 표시해줘.

1. 정의되지 않은 용어 — glossary에 없는데 본문에 쓰인 단어
2. 문서 간 모순 — 같은 항목이 서로 다른 값/규칙으로 적힌 곳
3. 빠진 상태 — 정상 외 로딩·빈·에러·권한·오프라인 정의 여부
4. 빠진 범위 — Out of scope에 명시 안 된 모호한 영역
5. 검증 불가 표현 — '빠르게·자연스럽게·직관적으로' 같은 막연한 말
6. 빠진 비기능 — 성능·접근성·보안·다국어·데이터 보관/삭제
7. 추적 끊김 — REQ ↔ SCR ↔ TC 매핑이 빠진 항목

각 지적은 구체적 위치(파일·줄)와 함께 알려줘.
```

## 재사용 5가지 운영 패턴 (원리와 별개 — 실전 운영 노하우)
1. **맥락을 일감 옆에 둔다** — 규칙·맥락을 한 저장소에. 구두 지시 최소화.
2. **금지부터 정한다** — "하라"보다 "하지 마라". 이유까지.
3. **검증을 1초로** — 검증 가능한 부분을 떼어 AI가 스스로 증명 (`swift test` · `scripts/check-docs.sh` 류).
4. **사람/AI 경계 명문화** — 콘텐츠·디자인·확인 = 사람 / 구현·반복·검증 = AI.
5. **기억을 지속** — DEVLOG·HISTORY로 세션 간 맥락 복원.
